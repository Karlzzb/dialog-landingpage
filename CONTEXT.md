# 领域术语表（产教融合专家助理）

本文件只维护领域语言（glossary），不放实现细节。

## 顶层架构

- **确定性骨架（Deterministic Skeleton）**：LangGraph 中的薄外层图。
入口负责初始化、用户画像、权限落库、意图分流；出口负责质检门（PII 脱敏、数字溯源、注入拦截、外部数据隔离、优雅兜底）。
这些是图的硬约束，不依赖 LLM"自觉"。

- **ReAct 内核（ReAct Core）**：骨架中间的"规划 → 调工具 → 观察 → 再决策"自主循环。
取代 bisheng 里用 condition 硬接的流水线。
推理与工具编排在此自由进行。

## 意图与流程

- **对话流（Chat Flow）**：寒暄、问候、情绪宣泄、主观观点、与领域无关的开放互动。
目的是拟人化沟通反馈，不触发检索。

- **知识流（Knowledge Flow）**：用户查询客观存在、有据可查的政策/制度/标准/数据现状/名词定义。
目的是获取确切事实，会触发数据源检索。

> 分流机制：不设入口硬分类器。对话流/知识流的判断由 ReAct 内核在推理时自主完成（"要不要调检索工具"）。
> 混合意图与多轮意图漂移天然处理。

> 对话流行为约束（≤50字/禁越界/专家口吻/尾部业务引导拉回）不设出口质检门，靠内核 system prompt + few-shot 保证。
> 仅"≤50字"这条可确定性度量的，加一次纯代码后处理截断/压缩作廉价兜底。
> 闲聊不是独立图分支，而是内核"判断不需工具→按对话流规则直接作答"的自然出口。

## 会话隔离与权限

- **会话隔离（Session Isolation）**：当前阶段仅靠 **session id 的唯一性**保证不同用户会话互不干扰。
session id 对应 LangGraph 的 thread_id / checkpointer 线程。

- **跨轮持久化**：用 LangGraph Checkpointer 持久化整个 State（含会话记忆/滚动实体摘要），thread_id = session id。
改写步读会话记忆、每轮末更新，全在图内闭环；后端每轮只传 session id + 本轮 user input。
后端用 **Redis**；会话记忆设 TTL 避免无限增长（TTL 取值待定）。
富权限模型（角色/可见范围/脱敏）**暂不设计**，待业务真正引入角色时再启用 PermissionScope 方案。

- **PermissionScope（权限范围）· 未来项**：设计目标是 step1 算出的结构化会话级不变量（角色 + 可见范围 + 脱敏级别），由统一权限注入层在查询执行前转成行级过滤 + 脱敏规则，模板不碰权限。
当前不落地（见 ADR 0005 状态说明）。

## 状态与记忆

- **会话记忆（Session Memory）**：跨轮持久的结构化状态，追求 context 最小。
包含：用户画像（姓名/角色/组织）、权限范围、以及滚动维护的"关键实体/约束摘要"（如 地域=武汉市、主题=校企合作）。
不全量保存原始对话历史。

- **自包含查询（Self-contained Query）**：每轮进内核前，由改写步基于会话记忆把用户输入补全成不依赖上下文的完整查询。
例：第2轮"江夏区有额外的吗？" → "武汉市江夏区校企合作补贴"。
改写只做指代消解，不做流程分叉（与"不设入口硬分类器"不冲突）。

- **单轮 State**：本轮用完即弃——当前自包含查询、覆盖度表、各数据源原始返回、草稿结论。

## 模型分工

- **两档模型分层**：
  - **强模型 = kimi 2.6**：ReAct 规划循环 + 工具编排 + 结论生成（工具调用稳、长上下文装 Evidence，内核命脉）。
  - **快模型 = deepseek-v4-pro**：问题改写/指代消解 + 对话流闲聊（快、中文好、简单任务够用）。
  - **qwq-plus（可选）**：默认不用；仅当"信息单元拆解/覆盖度评估"需更深推理时，可把该子步单独换成它。不进主 ReAct 循环（重推理模型在高频工具循环里易卡壳）。
- **配置**：三模型共用同一 OpenAI 兼容 `base_url` + `key`（.env 引用，真实值由用户填），按角色只切换 model 名（如 `STRONG_MODEL` / `FAST_MODEL`）。

## loop 终止与失败

- **退出路径**：内核 loop 有三条退出——
  1. 覆盖完成（`remaining_units == 0`）→ 正常强制作答。
  2. 穷尽仍缺失 → 判定不可解，走盲区路径。
  3. 安全阀触发 → 兜底强制作答。

- **盲区（Blind Spot）**：loop 的一等状态（非异常）。
知识库→数据库→联网三层穷尽后仍未覆盖的信息单元。
结论生成时须坦诚告知边界、给补救方向（对上 T19/E2E-02 的能力要求）。

- **局部降级（Per-source Degradation）**：单个数据源超时/5xx 时自动重试 1 次（退避 500ms），仍失败则放弃该源、在覆盖度表标记"已尝试·不可用"，loop 继续走剩余层。
单源失败不阻断整轮；仅当某单元的所有可覆盖源都失败，该单元才落入盲区。
技术堆栈绝不外泄，兜底话术人性化。

- **安全阀（Safety Caps）**：最大工具调用 6 次 / 最大迭代 8 步（初值，上线后按真实分布调）。

## 结论生成

- **结论生成（Final Answer）**：ReAct 内核的终止动作，不设独立节点。
loop 终止时强制 LLM 直接产出最终回复（关闭工具调用，逼它基于 state 已有素材作答，不再检索）。
- 依赖 state 里的**分源结构化素材**（带来源标签 + 原始字段）：内部数据直接引用、外部数据自动带"据互联网公开信息"、数字直接取自 JSON 字段。
内外部隔离、盲区坦诚、来源标注是此步的输出行为，由素材结构 + prompt 自然导出（非事后质检）。
- **输出方式分场景**：结论生成走流式（长文本，体验收益大）；对话流闲聊整段返回（短，便于 ≤50 字后处理截断兜底）。

## 数据与检索

- **信息单元（Information Unit）**：把用户问题拆解出的最小信息需求，每个单元对应一条具体诉求。

- **数据源（Data Source）**：三类——
  - **INTERNAL_KNOWLEDGE 知识库**：校内规章制度、标准文件、已收录的国家红头文件。
  - **INTERNAL_DATABASE 数据库**：校内合作企业明细、学生实习状态、考勤率、对口率等统计数据。
  - **EXTERNAL_SEARCH 联网**：校内绝对没有的数据（企业工商/财报、外部新闻、未收录的实时政策等）。

- **Evidence（结构化证据）**：三个检索工具（知识库/数据库/联网）统一的返回契约。
`{ source_type, content, citation（文件/条款/URL/字段，可回溯）, raw（原始JSON，供数字溯源）}`。
结论生成据此天然做到内外部隔离、来源标注、数字取自 raw 字段。

- **联网内容过滤**：联网工具边界内置的**基础敏感词/合规过滤**（涉政/违法等）。
外部内容进入 state 前清洗/拦截，绝不把未过滤原文喂给 LLM（数据层职责，非独立质检节点）。

- **知识库工具**：对接 bisheng 知识库后端（`http://10.30.186.171:3001`，Cookie+JWT 鉴权，响应信封 `{status_code, status_message, data}`），返回统一 Evidence 契约。
需要的是**检索(retrieval)端点**（query → 相关 chunk + 出处）——`bisheng_rag_api.py` 只覆盖了入库侧，检索端点待提供，是本工具的阻塞依赖。

- **知识库入库管线（Ingestion）**：`bisheng_rag_api.py` 提供的建库/上传/预览切分/正式入库/列库能力。
属于离线运维流程，不在 agent 运行时；仅作为知识库如何被建立与灌入的背景。

- **业务查询能力（Query Capability）**：数据库工具的最小单元。
一个预定义、带参数、带权限过滤、只读的安全查询（如"查某学院某专业实习对口率"）。
LLM 只选能力 + 填参数（`{capability, params}`），不写 SQL。
注入/越权/PII/只读在此层做结构性保证。

- **命中即停（Hit-and-Stop）**：按 知识库 → 数据库 → 联网 逐层查询；
当某层已满足全部剩余信息单元时，必须终止后续查询，不再向下穿透。

- **覆盖度表（Coverage Table）**：内核首步产出的结构化对象。
包含信息单元清单、每单元的数据源匹配、覆盖状态（已覆盖/仍缺失）。
每层查询后更新，`remaining_units == 0` 即触发命中即停。
是 T11-T13 确定性校验的载体。
