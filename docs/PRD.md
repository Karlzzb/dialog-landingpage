## Problem Statement

用户是职业教育领域的一线人员（教师、院系/校级业务人员）。
他们需要一个"产教融合专家助理"，既能日常寒暄式互动，也能回答有据可查的政策/制度/标准/数据现状类问题，还能查校内统计数据（对口率、考勤率、实习状态、薪资等）。
现有实现建立在 bisheng 之上：53 个节点、大量 condition 硬接成一条流水线。
它是"流程式"而非真正的智能体循环，能力低下——不能自主多步推理、不能按需编排工具、多轮上下文（尤其隐含限定词）频繁丢失、混合意图处理生硬。
结果是达不到业务要求。

## Solution

从用户视角看，助理应当：
- 听得懂就直接答，寒暄就简短礼貌地承接并把话题拉回业务。
- 遇到需要事实依据的问题时，自己判断该查什么、按 知识库 → 数据库 → 联网 的顺序逐层查、够了就停，不做无谓查询。
- 多轮对话里记得上文（"江夏区有额外的吗"能自动理解为"武汉市江夏区校企合作补贴"）。
- 给结论时清楚标注来源，内部数据直接引用、外部数据注明"据互联网公开信息"、数字忠于原始数据；查不到就坦诚说明边界并给下一步指引，绝不编造。
- 任何一步出错（某数据源超时/报错）都优雅降级，不把技术细节甩给用户。

技术上：用 LangGraph 重构为**确定性骨架 + ReAct 内核**。
薄的确定性外层管入口（改写/记忆）与出口（结论作答）；中间是真正的"规划 → 调工具 → 观察 → 再决策"循环，取代旧的 condition 流水线。

## User Stories

1. 作为一线教师，我想向助理问好时得到简短专业的回应，以便快速进入正题。
2. 作为业务人员，我想让助理判断我的问题是闲聊还是业务查询，以便它不会对寒暄误触发检索。
3. 作为业务人员，当我一句话里既有寒暄又有业务（"你好，帮我查下政策"），我想它先自然承接再进入查询，以便对话顺畅。
4. 作为一线教师，我想问某项政策/制度/标准"是什么/有没有/怎么规定"，以便获取确切事实。
5. 作为业务人员，我想查校内统计数据（对口率、考勤率、实习状态、平均薪资），以便支撑决策。
6. 作为业务人员，我想让助理把我的复杂问题拆成若干具体信息需求，以便每个诉求都被覆盖、不遗漏。
7. 作为业务人员，我想让助理只在必要时才查数据库/联网，以便结果快且不浪费。
8. 作为业务人员，我想让助理按 知识库 → 数据库 → 联网 逐层查询、命中即停，以便优先采信更权威的内部来源。
9. 作为一线教师，我想在多轮对话里被记住上文限定词（地域、主题、专业、班级、时间窗口），以便后续追问无需重复背景。
10. 作为业务人员，我想让助理把"江夏区有额外的吗"这类省略句自动补全成自包含查询，以便结论精准命中我的真实意图。
11. 作为业务人员，我想在结论里看到每条核心结论的来源标注，以便判断可信度。
12. 作为业务人员，我想让内部数据与外部联网数据在回复中严格隔离、外部数据带弱化提示，以便不被未验证信息误导。
13. 作为业务人员，我想让回复里的每个关键数字都能追溯到底层数据，以便不被数字幻觉误导。
14. 作为业务人员，当数据查不到时，我想得到坦诚的边界告知与替代/下一步指引，而不是被编造的答案，以便安全决策。
15. 作为业务人员，当某个数据源临时不可用时，我想得到"基于其余源生成"的结论并被告知该源不可用，以便理解结论的局限。
16. 作为业务人员，我想在任何异常下都看到人性化的兜底话术而非技术堆栈，以便体验不被破坏。
17. 作为连续闲聊的用户，我想每轮闲聊回复都简短（≤50 字）且尾部带业务引导，以便被温和地拉回正题。
18. 作为用户，我想我的会话与其他用户完全隔离，以便隐私与上下文不串台。
19. 作为用户，我想跨轮对话被持久记住（在会话生命周期内），以便中断后仍能接续。
20. 作为提出复杂多源问题的用户（"A 企业财报如何？我校在该企业对口率多少？"），我想两个信息单元都被覆盖、且财报与对口率被隔离标注，以便一次拿到完整可信的答案。
21. 作为提出隐性对比意图的用户（"这两个专业就业率"），我想不仅拿到数据还得到对比分析，以便结论可直接用于决策。
22. 作为运维/开发者，我想通过 .env 配置模型（同一 base_url/key、按角色切换 model 名），以便零改码切换或升级模型。
23. 作为开发者，我想数据库查询是预定义的参数化能力而非自由 SQL，以便杜绝注入并保证只读。
24. 作为开发者，我想知识库工具对接可插拔的 RAG API 适配层，以便后续接入真实 RAG 接口而不改主体。
25. 作为开发者，我想三个检索工具都返回统一的结构化 Evidence（来源/可回溯/原始 JSON），以便结论生成天然做到标注与溯源。
26. 作为开发者，我想联网工具在边界内置基础敏感词/合规过滤，以便外部原文绝不裸奔进 LLM。
27. 作为开发者，我想 ReAct 循环有安全阀（最大工具调用/迭代步数），以便不会死循环烧 token。
28. 作为开发者，我想把质检/评分体系留在 agent 之外，以便 agent 职责单一、关键路径延迟低。
29. 作为业务负责人，我想新增一类业务查询只需增加一个查询能力模板，以便扩展可控可维护。
30. 作为用户，我想长结论以流式逐字呈现、短闲聊整段返回，以便兼顾等待体验与字数可控。

## Implementation Decisions

**总体架构（ADR 0001）**
确定性骨架 + ReAct 内核，技术栈 LangGraph。
外层确定性节点负责入口（改写/记忆）与出口（结论作答）；内层是带 conditional loop 回边的真正 ReAct 循环，负责规划、调工具、观察、再决策。
不设入口意图硬分类器——对话流/知识流的判断由内核自主完成（"要不要调检索工具"）；闲聊是内核"判断不需工具→直接作答"的自然出口，而非独立图分支。

**入口：会话记忆与改写（ADR 0003）**
每轮进内核前有一个改写步（快模型）：基于结构化会话记忆，把用户输入补全成自包含查询（消解指代，不做流程分叉）。
会话记忆是跨轮持久的结构化状态，只维护滚动的"关键实体/约束摘要"，不保存全量对话历史，追求 context 最小。

**检索规划：覆盖度表驱动的渐进检索（ADR 0002）**
内核首步产出结构化《覆盖度表》：信息单元清单 + 每单元的数据源匹配 + 覆盖状态。
按 知识库 → 数据库 → 联网 逐层查询；每层查完更新覆盖度表；`remaining_units == 0` 即命中即停、跳出，不向下穿透。
覆盖度表是动态的，可随观察补充新单元、修正数据源匹配。

**工具层**
- 三工具统一返回 **Evidence** 契约（ADR 0007）：`{ source_type, content, citation, raw }`。
- 数据库工具 = 参数化查询能力集（ADR 0004），非 Text2SQL；LLM 只输出 `{capability, params}`，SQL 人写死、只读、参数强类型校验。新增业务查询 = 新增一个能力模板。
- 知识库工具 = 对接 bisheng 知识库后端的适配层，返回 Evidence（详见下方"知识库 RAG 接入"）。
- 联网工具 = 边界内置基础敏感词/合规过滤，过滤后才进 State。

**结论生成**
不设独立节点；ReAct 内核在 loop 终止时**强制作答**（关闭工具调用，逼模型基于 State 已有素材直接答）。
依赖 State 里分源的 Evidence：内部数据直接引用、外部数据自动带"据互联网公开信息"、数字取自 `raw` 字段。
内外部隔离、盲区坦诚、来源标注是此步的输出行为，由素材结构 + prompt 导出。

**loop 终止与失败**
三条退出路径：覆盖完成 / 穷尽仍缺失（盲区，一等状态，坦诚告知边界+给指引）/ 安全阀（最大工具调用 6 次、最大迭代 8 步，初值可调）。
局部降级：单源超时/5xx 自动重试 1 次（退避 500ms），仍失败则放弃该源、在覆盖度表标记"已尝试·不可用"，loop 继续；仅当某单元所有可覆盖源都失败才落入盲区。
技术堆栈绝不外泄，兜底话术人性化。

**会话与持久化**
会话隔离仅靠 session id 唯一性（当前阶段，富权限模型暂缓，见 ADR 0005）。
跨轮用 LangGraph Checkpointer 持久化整个 State，thread_id = session id，后端用 **Redis**；会话记忆设 TTL（取值待定）。
后端每轮只传 session id + 本轮 user input。

**模型分工**
两档，共用同一 OpenAI 兼容 base_url + key（.env 配置，真实值由用户填），按角色切换 model 名：
- 强模型 = **kimi 2.6**：ReAct 规划循环 + 工具编排 + 结论生成。
- 快模型 = **deepseek-v4-pro**：问题改写 + 对话流闲聊。
- **qwq-plus** 可选：仅当信息单元拆解/覆盖度评估需更深推理时替换该子步；默认不用，不进主循环。

**输出方式**
结论生成流式；对话流闲聊整段返回（便于 ≤50 字纯代码后处理截断兜底）。

## 知识库 RAG 接入（补充：基于 `bisheng_rag_api.py`）

**后端事实**：知识库承载于 bisheng 实例 `http://10.30.186.171:3001`。
鉴权方式为 HTTP 请求头 `Cookie` 携带 `access_token_cookie=<JWT>`（外加 `lang=zh-Hans`）。
所有响应遵循统一信封：`{ "status_code": 200, "status_message": "...", "data": {...} }`；业务成功需同时满足 HTTP 200 且 `status_code == 200`。

**`bisheng_rag_api.py` 覆盖的是"入库/管理"侧，不是"检索"侧**：
- `create_knowledge_base` → `POST /api/v1/knowledge/create`（建库）。
- `upload_file` → `POST /api/v1/knowledge/upload/{knowledge_id}`（上传，返回 presigned `file_path`）。
- `preview_file` → `POST /api/v1/knowledge/preview` + `get_preview_result` → `GET /api/v1/knowledge/preview/status`（预览切分，轮询至 `completed`）。
- `process_file` → `POST /api/v1/knowledge/process`（按 separator/chunk 参数正式切分入库；状态码 5 处理中 / 6 成功 / 7 失败）。
- `list_knowledge_bases` → `GET /api/v1/knowledge`（分页列库）。
- `analyze_chunks`（本地，调用 LLM 评估切分深度/重合度，属离线切分调参，与 agent 运行时无关）。

**关键缺口**：该文件中**没有检索/问答端点**（输入用户 query、返回相关 chunk + 出处）。
agent 的知识库工具正是需要这个检索调用来产出 Evidence。
因此 **bisheng 检索端点（如相似度检索 / QA 接口）是知识库工具的阻塞依赖，待用户提供**。
在其到位前，知识库工具适配层按 Evidence 契约先定义接口并用假实现打桩，主体图与其余工具不被阻塞。

**知识库工具适配层的目标契约**（检索端点到位后落地）：
- 输入：用户的自包含查询（可能附带目标知识库 id / 过滤条件）。
- 调用：bisheng 检索端点，携带 Cookie+JWT，解析统一信封。
- 输出：`Evidence[]`，其中 `source_type = INTERNAL_KNOWLEDGE`、`content = chunk 文本`、`citation = 文件名/知识库/chunk_index 等可回溯信息`、`raw = 原始 chunk JSON`。

**鉴权与密钥（必须整改，非可选）**：
- `bisheng_rag_api.py` 里**硬编码了 bisheng 的 JWT cookie 与火山引擎 LLM 的 api_key**——这是提交进仓库的明文密钥，须立即移入 `.env`、从版本库剔除。
- 该 JWT 的 `exp` 为 **2026-07-10 05:44 UTC（即将过期）**，说明 bisheng 用短期 token。
agent 侧知识库工具必须有**长效鉴权策略**（可续期的凭据/账号登录换 token/服务端代理），不能依赖手工粘贴的过期 token。
- bisheng 入库脚本内自带的 LLM（`deepseek-v3-2`，火山引擎）与本 agent 的两档模型是**相互独立**的两套配置，不要混用。

**入库（Ingestion）定位**：建库/上传/切分/入库是**离线运维流程**，不在 agent 运行时范围。
它解释了知识库如何被建立与灌入，可作为运维脚本保留，但不是本 PRD 要实现的 agent 能力。

## Testing Decisions

**什么是好测试**：只测外部行为，不测实现细节。
断言"给定输入 + 桩化的工具/LLM，产生什么最终回复与 State"，而非断言内部函数被如何调用。

**主接缝（首选、唯一、最高）**：编译后的 LangGraph 图入口。
以 `invoke(session_id, user_input)` 驱动，注入假工具（RAG/查询能力/联网）与桩化 LLM 输出，断言：命中即停顺序、覆盖度表演化、盲区路径、来源标注与内外部隔离、单源失败降级、闲聊 ≤50 字与业务引导、多轮自包含改写（用 Redis checkpointer 的测试实例跨两轮验证）。
LLM 用可编排的桩（预置每一步的返回/工具调用），使图逻辑可确定性测试。

**第二接缝**：工具适配层（纯函数式，脱离图独立测）。
- 查询能力层：给定 `{capability, params}`，断言只读、参数校验、注入输入被拒。
- 联网过滤：给定含敏感词的外部内容，断言被清洗/拦截、绝不透传。
- RAG/联网适配：断言外部返回被正确规整为 Evidence 契约。

**测试模块**：图入口的行为测试；工具适配层的单元测试。

**先行样例**：当前仓库尚无测试。
新建测试须遵循"高接缝 + 桩化外部依赖"的模式，作为后续所有测试的先行样例。

**明确不测**：LLM 的真实生成质量（意图分流准确率、结论优劣、口吻等判断项）。
这些属于独立质检/评测体系（ADR 0006），不在本 PRD 的测试范围。

## Out of Scope

- 质检/评分体系（design.md 第二部分 T1–T28 的打分、通过门槛、E2E 评分、离线标注、告警）——独立系统，把本 agent 当黑盒评估（ADR 0006）。
- 富权限模型（角色/可见范围/脱敏分级、PermissionScope、统一权限注入层）——当前仅 session id 隔离，权限方案作为未来项保留（ADR 0005）。
- 后端鉴权与用户画像入参契约的详细设计。
- RAG 的分块/嵌入/索引策略（对接外部 RAG API，其内部实现不在此）。
- 具体业务查询能力目录的枚举（需业务侧提供清单）。
- 可观测性/日志/告警的完整方案。

## Further Notes

**待外部输入的挂起项（不阻塞主体搭建，均设计为留好接口的可插拔适配层）**：
1. **bisheng 知识库检索(retrieval)端点** → 定知识库工具适配层（`bisheng_rag_api.py` 只给了入库侧，检索侧缺；见"知识库 RAG 接入"）。
2. 业务查询能力目录（对口率/考勤率/实习状态/薪资…）→ 业务侧枚举。
3. 会话记忆 TTL 具体取值。
4. 模型的真实 base_url / key（.env 填写）。
5. **bisheng 长效鉴权凭据**（现有 JWT 2026-07-10 过期，硬编码密钥须移入 .env 并从版本库剔除）。

**对齐要点**：全程中文交互。
安全属性（只读、（未来的）权限、PII 脱敏）是数据层的固有性质，由查询能力层与（未来的）权限注入层结构性保证，而非事后质检门。
