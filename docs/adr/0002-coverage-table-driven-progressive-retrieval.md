# ADR 0002：覆盖度表驱动的渐进检索

状态：已接受
日期：2026-07-09

## 背景

`design.md` 对检索规划提出了性质冲突的要求：
- T11（信息单元拆解）、T12（单元→数据源匹配）、T13（覆盖度评估）是**确定性条目**，必须显式做、可检验。
- T14（渐进查询、命中即停）是**判断项**，要求逐层执行、每层判定缺失、满足即停——一个动态循环。

纯自由 ReAct 无法结构化保证 T11-T13；纯流水线又违背 ReAct 内核的目标。

## 决策

内核采用 **显式规划 + 结构化覆盖度表驱动的渐进循环**（介于 plan-and-execute 与 ReAct 之间）：

1. 内核首步强制产出结构化《覆盖度表》：信息单元清单 + 每单元的数据源匹配 + 覆盖状态。
2. 按 知识库 → 数据库 → 联网 逐层查询；每层查完回来更新表中各单元的"已覆盖/仍缺失"。
3. 循环条件：`remaining_units == 0` 即命中即停，跳出，不向下穿透（满足 T14）。
4. 覆盖度表是动态的：可随观察补充新单元、修正数据源匹配——保持 ReAct 特性，而非一次性静态分叉。

## 备选方案

- **纯 ReAct 自由发挥**：T11-T13 确定性无法结构化保证——被否。
- **三工具 + prompt 约定顺序**：顺序与命中即停全靠模型自律，脆弱——被否。

## 影响

- T11-T13 从"事后检文本"变为"state 里的结构化对象"，覆盖度表存在即可被确定性校验。
- 命中即停成为干净的循环条件，不靠模型自觉。
- 需精心设计覆盖度表的 schema 与更新 prompt。
