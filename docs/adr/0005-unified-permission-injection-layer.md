# ADR 0005：统一权限上下文注入层

状态：已接受为目标架构，当前阶段暂缓落地
日期：2026-07-09

> 补充（2026-07-09）：当前阶段会话隔离仅靠 session id 唯一性，业务尚未引入角色/权限分级。
> 本 ADR 作为将来引入权限时的目标方案保留，现在不实现 PermissionScope 与注入层。

## 背景

E2E-01（权限边界铁律）是一票否决，T27（越权拦截）是确定性条目。
行级权限（普通教师看本班脱敏 / 院系管理员看本院 / 校领导看全校）若散落在各查询能力模板里，漏一处即失守，且维护困难。

## 决策

- 权限过滤走**统一注入层**，权限逻辑单点强制、单点审计。
- step1 初始化时算出结构化 `PermissionScope`（角色 + 可见范围 + 脱敏级别），存入会话记忆（会话级不变量，呼应 T1 不重复初始化）。
- 所有业务查询能力执行前，由统一执行器强制把 `PermissionScope` 转成行级过滤条件 + 脱敏规则注入；能力模板本身不碰权限。
- 数据库侧行级安全（RLS）作为可选的纵深防御，视基础设施而定，不作为唯一防线、不阻塞设计。

## 备选方案

- **每模板各写权限过滤**：逻辑散落、易漏、维护噩梦——被否。
- **仅靠数据库 RLS**：落地门槛高、依赖基础设施——不作为唯一防线。

## 影响

- 模板作者不可能"忘记加过滤"，E2E-01 由结构保证。
- 需设计 `PermissionScope` → 过滤/脱敏 映射规则；能力模板须按统一约定暴露可过滤维度（dept/class 等）。
