# ADR 0004：数据库工具用参数化查询能力集，而非 Text2SQL

状态：已接受
日期：2026-07-09

## 背景

数据库查询工具压着多条确定性红线：T22（注入防护、账号只读）、T27（权限越权拦截）、T26（PII 脱敏）、T25（数字反向溯源）。
开放式 Text2SQL 表达力最强，但注入面最大、越权最难防、易生成错 SQL / 幻觉字段，这些红线几乎不可能 100% 保证。

## 决策

数据库工具做成 **参数化查询能力集（语义层）**：
- 预定义一组业务查询能力（如"查某学院某专业实习对口率"、"查某班级脱敏平均薪资"），每个是带参数、带权限过滤、只读的安全查询。
- SQL 由人写死，只有参数来自 LLM；LLM 的工作退化为 `{capability, params}` 的结构化工具调用。
- 注入、越权、PII 脱敏、只读在模板层做结构性根治，参数强类型校验。

## 备选方案

- **开放式 Text2SQL**：确定性红线无法保证——被否。
- **Text2SQL + 受限视图**：越权风险降低但注入/幻觉字段仍在——被否。

## 影响

- 业务查询是有限集合（对口率/考勤率/实习状态/薪资……），穷举为能力清单可行且可维护。
- 代价：新增业务查询需人工加模板，不能零成本应对任意新问题——鉴于红线分量，接受此代价。
- 权限过滤如何组织（每模板各写 vs 统一注入层）见后续决策。
